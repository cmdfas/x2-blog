## 背景

搭建博客之前，我纠结过很久。

一开始我想过全部自研，前端、后端、管理后台、评论系统、图片上传、搜索、部署，全都自己做一遍。作为前端工程师，这当然很有吸引力，既能练技术，也能做成完全符合自己习惯的东西。

但冷静下来想想，我真正想要的并不是一个复杂的博客系统，而是一个可以长期写作、长期维护、长期迁移的地方。

博客的核心应该是内容，而不是系统本身。如果为了搭博客先花大量时间造轮子，很可能最后系统还没写完，想写的东西已经没了热情。

所以我给自己定了几个简单的目标：

- 使用 Markdown 写文章；
- 不依赖数据库；
- 尽量静态化，方便部署；
- 构建速度要快；
- 主题可以定制；
- 后续迁移成本要低；
- 国内外访问都要尽量稳定。

带着这些需求，我开始对比 Hugo、Hexo、WordPress 和 Next.js。

## WordPress：功能强，但对我来说太重了

WordPress 是非常成熟的博客系统。

它的优点很明显：

- 后台管理完善；
- 插件生态丰富；
- 主题非常多；
- 评论、用户、媒体管理等功能都很完整；
- 非技术用户也能比较快上手。

如果是做一个多人维护的网站，或者需要后台编辑、权限管理、会员系统、复杂插件生态，WordPress 依然是很强的选择。

但我的需求没有那么复杂。

我只是想写文章，偶尔改改页面，放一些自己的记录和想法。为了这些需求去维护 PHP、数据库、后台安全、插件更新，感觉有点过度设计。

而且 WordPress 动态站点还要考虑服务器环境、数据库备份、安全补丁、缓存优化等问题。虽然这些都可以解决，但它们会持续占用维护精力。

我希望博客尽量像一组静态文件：生成之后直接部署，坏了也容易恢复，迁移也只是复制文件和重新构建。

所以 WordPress 最后被我排除了。

## Hexo：也不错，但我更想要 Hugo 的简单和速度

Hexo 是很多人搭博客时的第一选择。

它的优点也很明显：

- 使用 Markdown 写作；
- 主题和插件生态丰富；
- 中文资料很多；
- 对前端工程师比较友好；
- 部署到 GitHub Pages 等平台也很方便。

如果只是想快速搭一个博客，Hexo 完全够用。

但我最后没有选择 Hexo，主要有两个原因。

第一个原因是依赖。

Hexo 基于 Node.js 生态，插件、主题和构建链路通常会带来一批 npm 依赖。作为前端工程师，我当然不陌生这些东西，但也正因为太熟悉，所以知道它们长期维护起来可能会有一些额外成本。

几年后重新拉项目，可能会遇到 Node 版本、依赖版本、插件兼容性等问题。不是不能解决，只是我希望博客系统本身越稳定越好。

第二个原因是速度。

Hugo 使用 Go 编写，构建速度一直是它很突出的特点。对于个人博客来说，前期文章少的时候差异不明显，但我更看重长期维护。以后文章、图片、归档、标签越来越多，构建速度和稳定性都会影响写作体验。

Hexo 是一个不错的选择，只是 Hugo 更符合我想要的方向。

## Next.js：很强，但博客不一定需要这么强

作为前端工程师，我当然也考虑过 Next.js。

Next.js 的能力很强：

- 可以做静态生成；
- 可以做服务端渲染；
- 可以做 API；
- 可以接 CMS；
- 可以实现非常自由的交互和页面设计；
- 生态活跃，适合做复杂产品。

如果我要做的是一个内容产品、作品集平台、带账号系统的社区，或者一个需要大量交互的站点，Next.js 会很有吸引力。

但我的博客现阶段不需要这么复杂。

我不需要登录系统，不需要后台管理，不需要服务端接口，也不需要每个页面都做成应用。大部分页面只是文章、归档、标签、关于页面和一些静态内容。

用 Next.js 写博客当然可以，而且能做得非常精致。但这也意味着我要维护更多工程化配置、依赖、组件、样式方案和部署细节。

对我来说，现阶段最重要的不是技术栈有多现代，而是能不能长期稳定地写下去。

博客不是为了证明我会用什么框架，而是为了保存内容。

## Hugo：更接近我想要的博客形态

最后选择 Hugo，是因为它刚好满足了我对博客的核心需求。

### 1. 静态化，部署简单

Hugo 生成的结果就是静态文件。

这意味着我可以把它部署到很多地方：EdgeOne Pages、GitHub Pages、Cloudflare Pages、Vercel，甚至普通的静态服务器都可以。

没有数据库，没有后台运行进程，也不需要维护复杂的服务端环境。

对于个人博客来说，这种简单非常重要。

### 2. 构建速度快

Hugo 的构建速度很快，本地预览体验也很好。

写文章时，我希望改完 Markdown 后页面能马上刷新。如果每次预览都很慢，写作体验会被打断。

博客系统不应该成为写作的阻碍。

### 3. 内容迁移成本低

文章本质上就是 Markdown 文件。

即使以后我不再使用 Hugo，这些文章也可以迁移到其他静态站点生成器，或者导入到别的系统里。

这点对我很重要。

平台可以换，主题可以换，部署方式可以换，但内容应该尽量保持独立。

### 4. 配置足够简单

Hugo 的配置集中在 `config.toml` 里，站点标题、语言、主题、菜单、参数都比较清晰。

我的博客目前只需要少量配置就可以运行起来，后续想增加 CDN、菜单、Logo、评论、统计等功能，也可以慢慢加。

这符合我的节奏：先让博客跑起来，再逐步完善。

### 5. 主题生态够用

Hugo 的主题很多，我最后选择了 Hugo Book。

它原本是偏文档风格的主题，但用来做个人博客也挺适合：结构清晰、阅读体验好、页面不花哨，也方便后续定制。

我不想一开始就在视觉设计上花太多时间。先选择一个稳定的主题，把内容写起来，比反复纠结样式更重要。

## 我的最终判断

如果简单总结这几个选择，我会这样看：

| 方案 | 适合场景 | 我没有选择的原因 |
| --- | --- | --- |
| WordPress | 需要后台、插件、多人管理的动态网站 | 对个人静态博客来说太重 |
| Hexo | 想快速搭建博客，喜欢 Node.js 生态 | 依赖和长期维护成本略高 |
| Next.js | 需要高度自定义、强交互、应用化站点 | 对当前博客需求来说过于复杂 |
| Hugo | 想要快速、稳定、静态、低维护的博客 | 最符合我的需求 |

所以我最后选择了 Hugo。

不是因为 Hugo 在所有方面都最好，而是因为它最适合我现在的目标：

> 用尽量低的维护成本，搭一个可以长期写下去的博客。

## 给想搭博客的人的建议

如果你也想搭自己的博客，我觉得不用一开始就追求完美。

可以先问自己几个问题：

- 你是想写内容，还是想做一个复杂系统？
- 你能接受维护服务器和数据库吗？
- 你希望博客五年后还能轻松迁移吗？
- 你更在意主题效果，还是写作体验？
- 你有没有足够的时间维护依赖和插件？

如果你只是想长期写作，记录技术、生活和思考，那么静态博客会是一个很好的起点。

Hexo、Hugo、Astro、Next.js 都可以做博客，关键不是哪一个最流行，而是哪一个最符合你的使用习惯。

对我来说，这个答案就是 Hugo。

先写起来，比选一个“完美方案”更重要。
